You are working in the Orac repository.

Use high reasoning.

Goal:
Align the canonical Orac protocol schema, protocol documentation, runtime stream frames, voice/slave handling, and web display event expectations so that the protocol boundary is explicit, tested, and no longer drifting.

This task follows the architecture audit in architecture-qa-results.md and the recent Phase 1/Phase 2 hardening work. The previous hardening pass added contract tests and quick fixes only. Use those new tests as guardrails.

Scope:
- Focus only on protocol/schema/runtime/display alignment.
- Do not extract the Orac controller.
- Do not introduce an LLM provider registry.
- Do not refactor the voice loop state machine.
- Do not implement Home Assistant runtime behaviour.
- Do not add plugin execution policy in this task.
- Do not tidy unrelated files.
- Do not modify unrelated existing changes shown by git status.

Important current issue:
The audit found that protocol/README.md and protocol/orac_protocol/resources/json_schema/protocol.schema.json describe the schema as canonical, but current runtime stream frames use fields such as:
- payload.delta
- payload.chunk
- payload.voice_session_id
- payload.turn_id
- payload.usage
- model identity in meta

The schema and the actual frames consumed by slave, voice, and web display must be reconciled.

Phase 0 - Inspect before editing:
1. Read:
   - architecture-qa-results.md
   - protocol/README.md
   - protocol/orac_protocol/resources/json_schema/protocol.schema.json
   - src/controller/orac.py
   - src/view/slave.py
   - src/orac_voice/voice_loop_local.py
   - src/orac_voice/voice_events.py
   - web/orac-display/src/App.tsx
   - web/orac-display/README.md
   - tests/test_protocol_contract.py
   - tests/test_display_event_pipe.py

2. Identify:
   - every protocol frame type currently emitted by the backend
   - every frame type currently consumed by the slave client
   - every frame/event type currently consumed by the local voice loop
   - every event alias currently consumed by web/orac-display
   - which payload fields are required, optional, or legacy-compatible

3. Before changing files, produce a short implementation note in your working summary:
   - whether the runtime should change to match the schema
   - whether the schema should change to match the runtime
   - whether backward-compatible aliases are needed
   - the minimal safe path

Expected direction:
Prefer making the schema accurately describe the current working wire protocol, unless there is a very small, low-risk runtime correction that makes the implementation cleaner without breaking clients.

Phase 1 - Schema alignment:
Update protocol/orac_protocol/resources/json_schema/protocol.schema.json so it validates the current intended wire protocol for:
- stream_start
- text_delta
- text_chunk
- stream_end
- stream_error
- stream_cancelled
- tts_playback_started
- tts_playback_finished
- tts_playback_cancelled
- tts_playback_error
- voice_turn_complete
- orac.voice.cancel request
- orac.voice.cancel response

Rules:
- Keep compatibility with fields currently consumed by src/view/slave.py, src/orac_voice/voice_loop_local.py, and web/orac-display/src/App.tsx.
- If both canonical and legacy names exist, document that explicitly.
- Do not silently remove existing fields unless all consumers are updated and tests prove safety.
- Make required fields strict only where the runtime genuinely guarantees them.
- Keep optional fields optional where older clients may omit them.

Phase 2 - Runtime validation:
Ensure outbound stream frames are validated consistently where practical.

Specifically:
- Normal response frames should continue to validate.
- Stream frames should validate against the aligned schema.
- Validation failures should be logged clearly.
- Do not let validation fail open in production.
- Preserve the explicit development-only no-op behaviour controlled by ORAC_ALLOW_NOOP_PROTOCOL_VALIDATION.

Do not add heavy runtime overhead or invasive refactoring.

Phase 3 - Documentation:
Update protocol/README.md so it clearly states:
- the canonical schema location
- the current stream frame contract
- payload fields for text_delta and text_chunk
- the distinction between immediate UI text deltas and speech-oriented text chunks
- voice cancellation request/response shape
- TTS playback lifecycle frames
- compatibility notes for frontend aliases, if applicable

Update web/orac-display/README.md only if the frontend event aliases or consumed fields need clarification.

Update or add tests so the protocol boundary is covered by executable checks.
