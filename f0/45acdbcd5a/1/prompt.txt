Please perform an architectural boundary QA pass over the Orac plugin execution path.

Current state:
- Plugin manifests support execution_policy metadata.
- PluginExecutionPolicy and PluginPolicyDecision exist in src/model/plugin_execution_policy.py.
- PluginRouter evaluates policy before importing/executing plugin code.
- Informational read-only plugins such as weather are allowed by default.
- Device-control plugins such as Home Assistant require confirmation, are denied by default, and may be marked scaffold.
- Scaffold plugins must be denied even if request metadata claims confirmation or allow.
- Unknown action types must fail closed.
- PluginExecutionResult carries provenance.
- Orac includes plugin provenance in response metadata and assistant-turn persistence metadata.

Please audit and tighten the boundaries between:
1. plugin discovery;
2. routing/candidate selection;
3. execution policy evaluation;
4. plugin import/invocation;
5. provenance creation;
6. response metadata;
7. assistant-turn persistence.

Important architectural rules:
- Plugin discovery must not import plugin code.
- Plugin code must never make final allow/deny decisions for its own execution.
- Orac core owns policy, confirmation, routing, persistence, and provenance.
- The LLM may help interpret intent, but must not authorise device control or mutation actions.
- Denied, scaffold, unknown-action, and confirmation-required plugin actions must not import or execute plugin code.
- Scaffold denial must override any request metadata.
- Unknown action types must fail closed.
- Request-supplied confirmation metadata must not be treated as a final trusted user confirmation in the long-term architecture. If currently used for tests, document it as provisional and add a clear seam/TODO for a trusted confirmation broker/manager.

Deliverables:
1. Add or update tests proving the boundaries above.
2. Add a concise architecture note, for example docs/plugin-execution-boundaries.md, describing the intended flow and ownership boundaries.
