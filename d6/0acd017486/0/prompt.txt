Please harden the explicit-only internet retrieval MVP.

Context:
The retrieval MVP is structurally correct and tests pass, but QA found several issues that should be fixed before live use.

QA findings to address:
1. SourceFetcher only validates http/https before urlopen. It does not protect against SSRF/internal fetches.
2. SearXNG config is stored under [retrieval.searxng], but Orac._init_retrieval() reads keys from [retrieval] using dotted key names.
3. Explicit search prompts skip plugin routing even when retrieval is disabled, unavailable, or produces no grounding. This can lead to ungrounded answers after the user explicitly requested web search.
4. Retrieval schema has relationship columns but no foreign key enforcement.

Required changes:

1. Harden SourceFetcher URL safety.
   - Allow only http and https schemes.
   - Reject file, ftp, data, javascript and all other schemes.
   - Reject localhost and localhost-like hostnames.
   - Resolve hostnames before fetching.
   - Reject loopback, private, link-local, multicast, reserved and unspecified IP ranges.
   - Reject obvious metadata-service addresses such as 169.254.169.254.
   - Apply the same validation before fetching and after every redirect.
   - Do not allow urllib or any HTTP client to follow redirects without validating the redirect target first.
   - Add a small configurable maximum redirect count.
   - Return a safe FetchedSource failure record for blocked URLs rather than crashing the user turn.

2. Add conservative fetch limits.
   - Configurable timeout.
   - Configurable maximum response bytes.
   - Only process text/html, text/plain and application/xhtml+xml for now.
   - Unsupported content types should produce a fetch_status/error_message record, not an exception through the user turn.
   - Keep PDF parsing out of scope unless already implemented.

3. Fix SearXNG config wiring.
   - Read base_url and timeout_seconds from [retrieval.searxng].
