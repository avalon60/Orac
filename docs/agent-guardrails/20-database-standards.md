# Database Standards

## Purpose

This document defines Orac database standards for SQL, PL/SQL, schema topology, object ownership, grants, DDL organisation, naming, and data access rules.

These standards are guardrails for human developers and coding agents.

They are intentionally strict.

If a change conflicts with this document, stop and ask for explicit approval before continuing.

---

## Core rules

- Use lowercase reserved words in SQL and PL/SQL.
- Use 2-space indentation.
- Prefer explicit constraint names.
- Define primary keys in separate `alter table` statements.
- Keep existing surrogate keys as identity columns unless explicitly instructed otherwise.
- Do not silently remove optimistic locking columns.
- Do not alter schema ownership boundaries without explicit approval.
- Do not write directly to `orac_core` tables outside the approved API layer.
- Do not rename existing objects merely to improve naming consistency.
- Do not introduce public synonyms unless explicitly approved.
- Do not bypass the TAPI/API layer for convenience.

---

## SQL and PL/SQL style

Use lowercase reserved words.

Good:

```sql
create table orac_core.example_table
(
  example_id number generated by default on null as identity not null,
  example_name varchar2(100 char) not null,
  created_on timestamp default systimestamp not null,
  created_by varchar2(128 char) default coalesce(
               sys_context('apex$session', 'app_user'),
               sys_context('userenv', 'proxy_user'),
               sys_context('userenv', 'session_user'),
               user
             ) not null,
  updated_on timestamp,
  updated_by varchar2(128 char),
  row_version number default 1 not null
);

alter table orac_core.example_table add constraint exmp_pk
  primary key (example_id);
```

Avoid uppercase reserved words.

Bad:

```sql
CREATE TABLE orac_core.example_table
(
  example_id NUMBER NOT NULL
);
```

### Formatting rules

- Use 2-space indentation.
- Align related clauses where it improves readability.
- Keep `when ... then` on the same line in PL/SQL exception handlers.
- In PL/SQL `if` statements, put `then` on the following line, aligned with `if`.

Preferred PL/SQL style:

```sql
if l_count = 0
then
  raise_application_error(-20000, 'No matching row found.');
end if;
```

Exception handler style:

```sql
exception
  when no_data_found then
    null;
```

---

## Schema topology

Orac uses a least privilege schema topology.

The main schemas are:

| Schema | Purpose |
|---|---|
| `orac_core` | Owns core tables and indexes. |
| `orac_api` | Owns API views, table triggers, and TAPI packages. |
| `orac_code` | Owns business logic packages, business views, materialized views, and orchestration logic. |
| `orac_apx_pub` | Public database connection schema for APEX applications and client-facing database services. |
| `orac_plugin` | Database connection schema for plugin-facing database services. |

Additional plugin-specific schemas may exist where a plugin needs private database storage.

Plugin schemas must follow the plugin standards in:

```text
docs/agent-guardrails/50-plugin-standards.md
```

Security and privilege rules must follow:

```text
docs/agent-guardrails/60-security-and-risk.md
```

---

## Schema responsibilities

### `orac_core`

`orac_core` owns:

- tables
- indexes
- primary key constraints
- unique constraints
- check constraints
- foreign key constraints where appropriate

`orac_core` must not contain business logic.

`orac_core` must not contain application-facing views.

`orac_core` must not be used as a normal application connection schema.

---

### `orac_api`

`orac_api` owns:

- table API packages
- table API package bodies
- pass-through API views
- table triggers
- API-layer validation required to protect table operations

API views normally expose a direct projection of one `orac_core` table.

API views should not contain business logic, joins, aggregation, or application-specific transformations.

Table triggers that maintain standard audit columns or row versions belong in `orac_api`.

---

### `orac_code`

`orac_code` owns:

- business rule packages
- orchestration packages
- business logic views
- materialized views
- higher-level APIs used by applications or plugins

`orac_code` may read through `orac_api` views.

`orac_code` may modify data only through `orac_api` TAPI packages.

`orac_code` must not perform direct DML against `orac_core` tables.

---

### `orac_apx_pub`

`orac_apx_pub` is a public connection schema for APEX and client-facing database access.

It may be granted access to selected `orac_code` packages and views.

It may be granted access to selected `orac_api` views where APEX needs direct region-level access.

It must not access `orac_core` tables directly.

It must not contain broad business logic unless explicitly approved.

---

### `orac_plugin`

`orac_plugin` is the public connection schema for plugin-facing database services.

It may be granted access to selected `orac_code` packages and views.

It must not access `orac_core` tables directly.

It must not own plugin-private data tables unless explicitly approved.

Plugin-private data should normally live in plugin-specific schemas.

---

## Grant model

The intended grant flow is:

```text
orac_core -> orac_api -> orac_code -> orac_apx_pub
                                  -> orac_plugin
```

General grant rules:

- Grant required DML privileges on `orac_core` tables to `orac_api`.
- `orac_api` may receive grants with grant option where required to expose controlled API views or packages.
- Grant execute on required `orac_api` TAPI packages to `orac_code`.
- Grant select on required `orac_api` views to `orac_code`.
- Grant execute on selected `orac_code` packages to `orac_apx_pub` and `orac_plugin`.
- Grant select on selected `orac_code` views to `orac_apx_pub` and `orac_plugin`.
- Grant selected DML privileges on `orac_api` views to `orac_apx_pub` only where required by APEX behaviour.
- Do not grant direct access from public schemas to `orac_core`.

Use private synonyms only where they are part of the approved access pattern.

Do not introduce public synonyms unless explicitly instructed.

---

## Data access rules

Business logic packages in `orac_code` must insert, update, delete, merge, or upsert data only through TAPI packages in `orac_api`.

Direct DML against `orac_core` tables is not permitted.

Single-row fetches should normally use TAPI functions or approved API-layer accessors where available.

Set-based reads should use `orac_api` views or approved `orac_code` views.

All data access from `orac_code` must occur through:

- `orac_api` views
- `orac_api` packages
- approved `orac_code` packages or views

Direct access to `orac_core` tables from `orac_code`, `orac_apx_pub`, `orac_plugin`, or plugin schemas is not permitted.

---

## Dynamic SQL rules

Dynamic SQL must be avoided unless there is a clear need.

Where dynamic SQL is required:

- Use bind variables.
- Do not concatenate untrusted values into SQL.
- Do not let the LLM provide executable SQL.
- Do not let plugins provide executable SQL unless the capability has an explicit reviewed design.
- Validate object names against allowlists when dynamic object references are unavoidable.
- Do not use dynamic SQL to bypass TAPI or API-layer rules.

Security rules for generated SQL are defined in:

```text
docs/agent-guardrails/60-security-and-risk.md
```

---

## Database object creation scripts

As a rule, each database object or constraint should have its own script.

Exceptions:

- comments for a single table or view may be grouped in one comment script
- grants for a single object may be grouped where this improves maintainability
- seed data may be grouped by functional area where appropriate
- install driver scripts may call other scripts

Script file name stems should reflect the object being created.

Script files must use the `.sql` extension.

Examples:

```text
table/messages.sql
constraint_pk/messages_pk.sql
constraint_fk/messages_conversations_fk1.sql
constraint_ck/messages_role_ck.sql
index/messages_conversation_idx1.sql
comment/messages_comments.sql
package_spec/messages_tapi.pks
package_body/messages_tapi.pkb
```

Use the repository's existing file extension conventions where already established.

```text

```

Scripts should be placed in the correct directory.

Do not place scripts in a miscellaneous location merely because it is convenient.

If a new directory type is required, document it before using it.

## Naming principles

Relational object names use `_` as the separator.

Object names should be meaningful and specific.

Avoid generic names.

Do not use Oracle reserved words or keywords as object names.

Do not use quoted identifiers.

Do not invent table abbreviations.

Agents must check:

```text
docs/agent-guardrails/table-abbreviations.csv
```

before creating or renaming database objects.

If an abbreviation is missing, propose one and add it to `table-abbreviations.csv` before generating DDL.

Do not use generic names such as:

```text
pk_<table>
fk_<table>_<parent>
idx1
constraint1
```

unless explicitly instructed.

---

## Renaming

Do not rename existing objects to improve naming consistency.

Existing object names take precedence over new naming standards.

Renaming is only permitted when explicitly instructed.

Do not introduce synonyms or duplicate objects as a workaround for renaming.

If a name is inconsistent but already used, preserve it unless the user explicitly requests a rename.

---

## Table naming

Tables should use meaningful singular names.

Use the existing Orac naming style.

Do not introduce plural table names unless an existing project pattern requires it.

Where table prefixes or domain prefixes are already established, follow the existing project convention.

Examples:

```text
messages
conversations
plugin_registry
plugin_capabilities
```

Avoid vague names:

```text
data
items
records
details
```

---

## Table abbreviations

Table abbreviations are used for constraints and indexes.

Rules:

- Reuse the abbreviation from `table-abbreviations.csv`.
- Do not invent abbreviations silently.
- Keep abbreviations stable once used.
- Do not change abbreviations merely to improve readability.
- If an abbreviation is missing, propose and add one before generating DDL.

Example mapping:

```text
messages = msg
conversations = cnv
plugins = plg
```

Actual abbreviations must come from the CSV file, not from guesses.

---

## Column naming

Column names should be meaningful and specific.

Use `_` as the separator.

Avoid abbreviated column name units unless the abbreviation is already standard in the project.

Avoid trailing numeric suffixes where a clearer name exists.

Good:

```text
address_line1
created_on
message_type
conversation_id
```

Avoid:

```text
addr_1
dt
typ
val
```

---

## Standard columns

Mutable Orac tables should normally include these standard columns:

| Column | Datatype | Nullable | Default | Purpose |
|---|---|---:|---|---|
| `created_on` | `timestamp` | No | `systimestamp` | Database timestamp when the row was created. |
| `created_by` | `varchar2(128 char)` | No | current application/session user expression | User, job, or process that created the row. |
| `updated_on` | `timestamp` | Yes | none | Database timestamp when the row was last updated. |
| `updated_by` | `varchar2(128 char)` | Yes | none | User, job, or process that last updated the row. |
| `row_version` | `number` | No | `1` | Optimistic locking value incremented on update. |

Recommended `created_by` expression:

```sql
created_by varchar2(128 char) default coalesce(
             sys_context('apex$session', 'app_user'),
             sys_context('userenv', 'proxy_user'),
             sys_context('userenv', 'session_user'),
             user
           ) not null
```

Recommended `created_on` expression:

```sql
created_on timestamp default systimestamp not null
```

Recommended `row_version` expression:

```sql
row_version number default 1 not null
```

Do not remove standard columns from existing tables without explicit approval.

Do not remove `row_version` from mutable tables unless the table is explicitly externally owned or append-only and the exception is documented.

---

## Soft delete columns

Where soft delete is required, use explicit soft delete columns.

Recommended optional columns:

| Column | Purpose |
|---|---|
| `deleted_on` | Timestamp when the row was soft deleted. |
| `deleted_by` | User, job, or process that soft deleted the row. |
| `deletion_reason` | Optional reason for the soft deletion. |

API views should normally hide soft-deleted rows unless the view is explicitly intended to expose them.

TAPI packages should own the soft-delete behaviour.

Do not implement soft delete ad hoc in application-facing SQL.

---

## Datatypes

Use Oracle datatypes deliberately.

Rules:

- Do not use `varchar`.
- Do not use `char` unless there is a specific fixed-width requirement.
- Use `varchar2`.
- Specify character semantics for text columns, for example `varchar2(100 char)`.
- Use `number` for numeric values unless a more precise type is required.
- Use identity columns for surrogate numeric primary keys where already used.
- Use `timestamp` or `timestamp with time zone` where time precision matters.
- Use `date` only where date-only or legacy semantics are intended.
- Use native `json` datatype where supported by the target Oracle version.
- Use `clob` plus `is json` only where native `json` is not available or not appropriate.
- Use `xmltype` for XML data.

Examples:

```sql
message_type varchar2(30 char) not null
content json not null
tokens_used number
created_on timestamp default systimestamp not null
```

---

## Default values

Prefer `default on null` where a value should be supplied even when the insert explicitly provides `null`.

Example:

```sql
message_id number generated by default on null as identity not null
```

For audit timestamps, use the existing project convention:

```sql
created_on timestamp default systimestamp not null
```

Do not add defaults that hide missing required business data.

---

## Yes/No columns

Simple yes/no columns should normally use `Y` and `N`.

Column names should normally end with `_yn`.

Example:

```sql
active_yn varchar2(1 char) default 'Y' not null
```

Add a check constraint:

```sql
alter table orac_core.example_table add constraint exmp_active_ck1
  check (active_yn in ('Y', 'N'));
```

Do not use `Yes`/`No`, `true`/`false`, or `1`/`0` for yes/no columns unless an existing integration requires it.

---

## Code columns

Columns used for codes should normally be uppercase and constrained.

Recommended pattern:

```sql
alter table orac_core.example_table add constraint exmp_code_ck1
  check (regexp_like(example_code, '^[A-Z0-9_]+$'));
```

Use code columns for stable natural identifiers where appropriate.

Do not use unconstrained free text where a controlled code is required.

---

## Primary keys

Primary keys should be defined in separate `alter table` statements.

Do not define primary keys inline in the `create table` statement.

Preferred pattern:

```sql
create table orac_core.messages
(
  message_id number generated by default on null as identity not null,
  ...
);

alter table orac_core.messages add constraint msg_pk
  primary key (message_id);
```

Primary key constraint naming:

```text
{table_abbr}_pk
```

Example:

```text
msg_pk
```

---

## Surrogate keys

Surrogate key columns should normally use the table abbreviation plus `_id`.

Pattern:

```text
{table_abbr}_id
```

Example:

```text
msg_id
cnv_id
plg_id
```

Where existing Orac tables already use clearer full names such as `message_id` or `conversation_id`, preserve the existing style.

Do not rename existing surrogate key columns to satisfy a new naming preference.

Surrogate keys should remain identity columns where already used.

---

## Foreign keys

Foreign key columns should normally reflect the referenced table and column.

Preferred simple pattern:

```text
{referenced_table_singular}_id
```

Examples:

```text
conversation_id
plugin_id
llm_id
```

Foreign key constraint naming:

```text
{child_abbr}_{parent_abbr}_fk{seq_nr}
```

Example:

```text
msg_cnv_fk1
```

For multiple foreign keys from the same child table to the same parent table, increment the sequence number.

For self-referencing foreign keys, use the same abbreviation for child and parent.

---

## Cross-schema foreign keys

Cross-schema referential integrity requires `grant references`.

Example:

```sql
grant references on orac_core.conversations to orac_core;
```

Where schema `A` owns the parent table and schema `B` owns the child table:

```sql
grant references on a.parent_table to b;
```

The grant must be applied before the foreign key constraint is created.

Use cross-schema foreign keys deliberately.

Do not add cross-schema referential integrity without considering install order, ownership, and deployment complexity.

---

## Constraints

Constraint naming patterns:

| Constraint type | Pattern |
|---|---|
| Primary key | `{table_abbr}_pk` |
| Foreign key | `{child_abbr}_{parent_abbr}_fk{seq_nr}` |
| Check constraint | `{table_abbr}_ck{seq_nr}` |
| Unique constraint | `{table_abbr}_uk{seq_nr}` |
| Column check constraint | `{table_abbr}_ck{seq_nr}` |

Do not explicitly name `not null` constraints unless an existing project pattern requires it.

Use constraints to enforce data quality wherever possible.

Do not rely only on application logic for data quality.

---

## Indexes

Primary keys and unique constraints should be supported by indexes.

Foreign keys should normally be indexed where they are used in joins or where child-row locking could matter.

Index naming patterns:

| Index type | Pattern |
|---|---|
| Primary key index | same as primary key constraint |
| Unique index | `{unique_constraint}_idx` |
| Foreign key index | `{foreign_key_constraint}_idx` |
| Function-based index | `{table_abbr}_fidx{seq_nr}` |
| Bitmap index | `{table_abbr}_bidx{seq_nr}` |
| Other index | `{table_abbr}_idx{seq_nr}` |

Examples:

```text
msg_pk
msg_cnv_fk1_idx
msg_idx1
msg_fidx1
```

### Concatenated indexes

When creating concatenated indexes, choose column order deliberately.

The most selective and commonly filtered leading columns should normally come first.

Do not create concatenated indexes without considering how the leading columns will be used.

### Function-based indexes

Use function-based indexes carefully.

Avoid function-based indexes that depend on application package functions unless there is a reviewed design.

Common acceptable examples:

```sql
create index exmp_fidx1 on orac_core.example_table (upper(example_code));
```

### Bitmap indexes

Bitmap indexes are normally unsuitable for OLTP-style tables.

Use bitmap indexes only where the table and workload justify them.

---

## Views

Regular views must end with `_v`.

Do not prefix views with `v_`.

API views in `orac_api` should normally be simple pass-through projections over one `orac_core` table.

Business views in `orac_code` may include joins, calculations, and business rules.

Preferred pattern:

```sql
create or replace force view orac_api.messages_v as
select message_id
     , conversation_id
     , message_type
     , turn_index
     , role
     , content
     , tokens_used
     , meta
     , llm_id
     , created_on
     , created_by
     , updated_on
     , updated_by
     , row_version
from orac_core.messages;
```

Views should not use owner-qualified object names where private synonyms or direct ownership already provide the approved access path.

Do not embed application-specific logic in API views.

---

## Materialized views

Materialized views must end with `_mv`.

Do not prefix materialized views with `mv_`.

Materialized views belong in `orac_code` unless there is an explicitly approved reason to place them elsewhere.

Use materialized views for deliberate performance, aggregation, or pre-computation needs.

Do not use materialized views to bypass API access rules.

---

## Triggers

Table triggers that maintain audit columns, row versions, or API-layer data rules belong in `orac_api`.

Trigger names should include the table name or abbreviation and the trigger timing/action.

Suggested suffixes:

| Timing/action | Suffix |
|---|---|
| Before insert row | `_bi` |
| Before update row | `_bu` |
| Before insert or update row | `_biu` |
| After insert row | `_ai` |
| After update row | `_au` |
| After insert, update, delete row | `_aiud` |
| Compound trigger | `_ct` |

Follow the existing repository naming pattern where one exists.

Do not create triggers in `orac_core` unless explicitly approved.

---

## Packages

Package names should be meaningful and singular.

TAPI packages belong in `orac_api`.

Business logic packages belong in `orac_code`.

Consumer-specific packages belong in the relevant public/access schema only where needed.

Standalone procedures and standalone functions are discouraged.

Prefer packages for related functionality.

---

## Sequences

Use identity columns for surrogate keys where already used.

Do not introduce sequences for new surrogate keys unless there is a specific reason.

Where sequences are required, use a clear name based on the owning object.

Example:

```text
example_table_seq
```

---

## Comments

Tables and columns should have comments.

Comments should explain purpose, not merely repeat the name.

Bad:

```sql
comment on table orac_core.messages is
  'Stores messages.';
```

Better:

```sql
comment on table orac_core.messages is
  'Stores persisted conversation events used to reconstruct history and build runtime LLM context.';
```

Column comments should describe business meaning, expected values, and important constraints.

Comments for a table or view may be grouped in one script.

---

## Message history and context columns

The `orac.messages.role` and `orac.messages.message_type` columns must follow the behavioural rules defined in:

```text
docs/agent-guardrails/70-context-management.md
```

The database layer may enforce allowed values and allowed combinations.

The semantic rules for context assembly, tool-result visibility, summaries, stale context, and plugin provenance belong to the context-management guardrail.

Do not add message roles or message types ad hoc from application code.

---

## JSON columns

Use native `json` columns where the target Oracle version supports them.

JSON content must have a documented structure.

Do not use JSON as a dumping ground for data that should be relational.

Use JSON for:

- model payloads
- plugin result payloads
- flexible metadata
- external service payloads
- context fragments

Do not use JSON to avoid modelling stable relational data.

Do not store secrets in JSON content or metadata.

Security rules for secrets are defined in:

```text
docs/agent-guardrails/60-security-and-risk.md
```

---

## Seed data

Seed data scripts belong in `seed_data`.

Seed data should be deterministic.

Use stable codes rather than generated IDs where possible.

Seed data scripts should be repeatable where practical.

Do not write seed data that depends on an uncontrolled current environment state unless explicitly required.

---

## Installation scripts

Installation scripts should preserve schema boundaries.

Install scripts must not:

- drop unrelated objects
- grant broad privileges
- connect as highly privileged users unless explicitly required
- hide errors
- silently ignore failed DDL
- change ownership boundaries
- remove constraints without approval

Install scripts should be clear, ordered, and repeatable.

---

## Agent rules

When generating or modifying database code, agents must:

1. Read this document first.
2. Check `table-abbreviations.csv` before naming constraints or indexes.
3. Preserve existing object names unless explicitly told to rename.
4. Preserve identity surrogate keys where already used.
5. Preserve standard audit and row version columns.
6. Preserve schema boundaries.
7. Use TAPIs for DML from `orac_code`.
8. Avoid direct access to `orac_core` outside the approved API layer.
9. Use lowercase SQL and PL/SQL reserved words.
10. Use 2-space indentation.
11. Add or update comments where new tables or columns are created.
12. Add constraints for controlled values.
13. Add tests or install verification where appropriate.
14. Stop and ask before introducing a new schema, grant pattern, or naming convention.

---

## Relationship to CoE database design guidance

Orac database standards are influenced by CoE and Ace Foundation database design practices, especially:

- least privilege schema layering
- separation of table ownership, API access, and business logic
- generated or controlled table APIs
- explicit grants
- consistent naming
- database-enforced data quality

This file is the Orac-specific authority for this project.

Do not paste large converted CoE documents into this guardrail file.

If detailed CoE material is needed, place it in a separate reference document or link to it from project documentation.

---

## Final rule

The database model must preserve this separation:

```text
orac_core  = data ownership
orac_api   = controlled table access
orac_code  = business logic
public schemas = consumer-specific access
plugin schemas = plugin-private storage
```

Any code that bypasses this separation is a design defect.
