# Database Standards

## Purpose

This document defines database standards for SQL, PL/SQL, domain schema topology, object ownership, grants, DDL organisation, naming, and data access rules.

These standards assume Oracle Database and PL/SQL.

These standards are guardrails for human developers and coding agents.

They are intentionally strict.

If a change conflicts with this document, stop and ask for explicit approval before continuing.

---

## Core rules

- Use lowercase reserved words in SQL and PL/SQL.
- Use 2-space indentation.
- Prefer explicit constraint names.
- Define primary keys in separate `alter table` statements.
- Keep existing surrogate keys as identity columns unless explicitly instructed otherwise.
- Do not silently remove optimistic locking columns.
- Do not alter schema ownership boundaries without explicit approval.
- Do not write directly to `<DOMAIN>_CORE` tables outside the approved API layer.
- Do not rename existing objects merely to improve naming consistency.
- Do not introduce public synonyms unless explicitly approved.
- Do not bypass the TAPI/API layer for convenience.

---

## SQL and PL/SQL style

Use lowercase reserved words.

Good:

```sql
create table <DOMAIN>_CORE.example_table
(
  example_id number generated by default on null as identity not null,
  example_name varchar2(100 char) not null,
  created_on timestamp default systimestamp not null,
  created_by varchar2(128 char) default coalesce(
               sys_context('userenv', 'client_identifier'),
               sys_context('userenv', 'proxy_user'),
               sys_context('userenv', 'session_user'),
               user
             ) not null,
  updated_on timestamp,
  updated_by varchar2(128 char),
  row_version number default 1 not null
);

alter table <DOMAIN>_CORE.example_table add constraint exmp_pk
  primary key (example_id);
```

Avoid uppercase reserved words.

Bad:

```sql
CREATE TABLE <DOMAIN>_CORE.example_table
(
  example_id NUMBER NOT NULL
);
```

### Formatting rules

- Use 2-space indentation.
- Align related clauses where it improves readability.
- Keep `when ... then` on the same line in PL/SQL exception handlers.
- In PL/SQL `if` statements, put `then` on the following line, aligned with `if`.

Preferred PL/SQL style:

```sql
if l_count = 0
then
  raise_application_error(-20000, 'No matching row found.');
end if;
```

Exception handler style:

```sql
exception
  when no_data_found then
    null;
```

## Schema topology

The project uses a least-privilege domain schema topology.

The same topology may be repeated across multiple domains.

| Schema | Purpose |
|---|---|
| `<DOMAIN>_CORE` | Owns domain tables, indexes, and constraints. |
| `<DOMAIN>_API` | Owns pass-through views, table triggers, and table API packages. |
| `<DOMAIN>_CODE` | Owns business logic packages, procedures, functions, types, business views, materialized views, and orchestration logic. |
| `<DOMAIN>_<CONSUMER>_PUB` | Consumer/access schema granted selected `<DOMAIN>_CODE` resources. One or more may exist. |
| `<PLUGIN_SCHEMA>` | Optional plugin-private schema used only where a plugin needs private database storage. |

Security and privilege rules must follow:

```text
docs/agent-guardrails/60-security-and-risk.md
```

---

## Schema responsibilities

### `<DOMAIN>_CORE`

`<DOMAIN>_CORE` owns:

- tables
- indexes
- primary key constraints
- unique constraints
- check constraints
- foreign key constraints where appropriate

`<DOMAIN>_CORE` must not contain business logic.

`<DOMAIN>_CORE` must not contain application-facing views.

`<DOMAIN>_CORE` must not be used as a normal application connection schema.

---

### `<DOMAIN>_API`

`<DOMAIN>_API` owns:

- table API packages
- table API package bodies
- pass-through API views
- table triggers
- API-layer validation required to protect table operations

API views normally expose a direct projection of one `<DOMAIN>_CORE` table.

API views should not contain business logic, joins, aggregation, or application-specific transformations.

Table triggers that maintain standard audit columns or row versions belong in `<DOMAIN>_API`.

---

### `<DOMAIN>_CODE`

`<DOMAIN>_CODE` owns:

- business rule packages
- orchestration packages
- business logic procedures and functions
- business logic types
- business logic views
- materialized views
- higher-level APIs used by applications, integrations, or consumer schemas

`<DOMAIN>_CODE` may read through `<DOMAIN>_API` views.

`<DOMAIN>_CODE` may modify data only through `<DOMAIN>_API` TAPI packages.

`<DOMAIN>_CODE` must not perform direct DML against `<DOMAIN>_CORE` tables.

---

### `<DOMAIN>_<CONSUMER>_PUB`

`<DOMAIN>_<CONSUMER>_PUB` is a consumer/access schema.

One or more consumer/access schemas may exist for a domain.

Each such schema may be granted access only to the selected `<DOMAIN>_CODE` packages and views it requires.

Direct access to `<DOMAIN>_CORE` is not permitted.

Direct access to `<DOMAIN>_API` should be avoided unless there is a reviewed reason.

Consumer/access schemas must not become alternative business-logic layers.

--
### Auto Maintained Columns
The followingf columns should appear, by default, in all tables:
  created_by           varchar2(64 char)
  created_on           timestamp with time zone not null,
  updated_by           varchar2(64 char)
  updated_on           timestamp with time zone not null
  row_version          number not null,
The automaintained columns are managed by a biu table trigger.

### `<PLUGIN_SCHEMA>`

`<PLUGIN_SCHEMA>` is optional.

It exists only where a plugin requires private database storage.

It may own plugin-private tables, indexes, views, packages, and supporting artefacts.

It must not own project core tables.

It must not bypass approved APIs to access protected domain state.

## Grant model

The intended grant flow is:

```text
<DOMAIN>_CORE -> <DOMAIN>_API -> <DOMAIN>_CODE -> <DOMAIN>_<CONSUMER>_PUB
```

General grant rules:

- Grant required DML privileges on `<DOMAIN>_CORE` tables to `<DOMAIN>_API`.
- `<DOMAIN>_API` may receive grants with grant option where required to expose controlled API views or packages.
- Grant execute on required `<DOMAIN>_API` TAPI packages to `<DOMAIN>_CODE`.
- Grant select on required `<DOMAIN>_API` views to `<DOMAIN>_CODE`.
- Grant execute on selected `<DOMAIN>_CODE` packages to each `<DOMAIN>_<CONSUMER>_PUB` schema that requires them.
- Grant select on selected `<DOMAIN>_CODE` views to each `<DOMAIN>_<CONSUMER>_PUB` schema that requires them.
- Do not grant direct access from consumer/access schemas to `<DOMAIN>_CORE`.

Use private synonyms only where they are part of the approved access pattern.

Do not introduce public synonyms unless explicitly instructed.

## Schema dependency generation guardrails

Schema dependency scripts connect the schema layers without changing ownership.

When generating schema dependencies, use this model:

```text
source schema tree  -> generated dependency tree
<domain>_core       -> <domain>_core/grant
<domain>_api        -> <domain>_api/view and <domain>_api/grant
<domain>_code       -> <domain>_code/grant
```

Dependency generation rules:

- Treat source schema directories as read-only inputs.
- Write generated views and grants to a separate target tree.
- Do not use the source root itself, or a child of the source root, as the generated target root.
- Use the base domain code only, for example `chat`, not `chat_core`, `chat_api`, or `chat_code`.
- Use the base owner prefix only, for example `chat`, not `chat_core`, `chat_api`, or `chat_code`.
- Keep generated dependency files in the correct `view` or `grant` subdirectory.
- Make generated dependency output repeatable and safe to regenerate.
- Include Liquibase rollback clauses where the project uses Liquibase formatted SQL.
- Do not hand-edit generated dependency files when the correct fix is to change the source DDL or generator inputs and regenerate.

Generated API pass-through views:

- For each `<DOMAIN>_CORE` table, generate the corresponding `<DOMAIN>_API` view as `<table_name>_v`.
- API pass-through views must project explicit columns from the table DDL.
- Do not use `select *` in generated API views.
- Do not add joins, filters, aggregation, business logic, or application-specific transformations to generated API pass-through views.
- If the table DDL cannot be parsed safely, skip generation for that table and report the skip rather than guessing columns.

Generated dependency script naming:

| Generated dependency | Directory | File pattern |
|---|---|---|
| API pass-through view for a core table | `<domain>_api/view` | `<table_name>_v.sql` |
| Core table grant to API schema | `<domain>_core/grant` | `<table_name>_to_<api_owner>.sql` |
| API view grant to CODE schema | `<domain>_api/grant` | `<view_name>_to_<code_owner>.sql` |
| API view grant to parsing schema | `<domain>_api/grant` | `<view_name>_to_<parsing_schema>.sql` |
| API program unit grant to CODE schema | `<domain>_api/grant` | `<program_name>_to_<code_owner>.sql` |
| CODE program unit grant to parsing schema | `<domain>_code/grant` | `<program_name>_to_<parsing_schema>.sql` |
| CODE view or materialized view grant to parsing schema | `<domain>_code/grant` | `<object_name>_to_<parsing_schema>.sql` |

Generated grant rules:

- Grant `select, insert, update, delete` on each `<DOMAIN>_CORE` table to `<DOMAIN>_API` with grant option where the API layer must expose controlled views or packages.
- Grant `read` on each required `<DOMAIN>_API` pass-through view to `<DOMAIN>_CODE`.
- Generate `insert`, `update`, or `delete` grants on `<DOMAIN>_API` pass-through views only when the project has explicitly approved updateable API views as part of the API layer.
- Grant `execute` on required `<DOMAIN>_API` program units to `<DOMAIN>_CODE`.
- If a parsing schema is used, grant `read` on required `<DOMAIN>_API` views to the parsing schema.
- If a parsing schema is used, grant `select` on required `<DOMAIN>_CODE` views and materialized views to the parsing schema.
- If a parsing schema is used, grant `execute` on required `<DOMAIN>_CODE` program units to the parsing schema.
- Do not grant generated dependencies directly from `<DOMAIN>_CORE` to `<DOMAIN>_CODE`, parsing schemas, plugin schemas, or consumer/access schemas.
- Do not broaden generated grants beyond the object type and direction required by the topology.

Dependency generation must be reviewed or rerun when any of these source objects change:

- `<DOMAIN>_CORE` tables
- `<DOMAIN>_API` views
- `<DOMAIN>_API` packages, procedures, functions, or types
- `<DOMAIN>_CODE` views or materialized views exposed to a parsing schema
- `<DOMAIN>_CODE` packages, procedures, functions, or types exposed to a parsing schema

Generated dependency scripts are part of the security boundary. A missing grant is an install defect; an excessive grant is a security defect.

## Data access rules

Business logic packages in `<DOMAIN>_CODE` must insert, update, delete, merge, or upsert data only through TAPI packages in `<DOMAIN>_API`.

Direct DML against `<DOMAIN>_CORE` tables is not permitted.

Single-row fetches should normally use TAPI functions or approved API-layer accessors where available.

Set-based reads should use `<DOMAIN>_API` views or approved `<DOMAIN>_CODE` views.

All data access from `<DOMAIN>_CODE` must occur through:

- `<DOMAIN>_API` views
- `<DOMAIN>_API` packages
- approved `<DOMAIN>_CODE` packages or views

Direct access to `<DOMAIN>_CORE` tables from `<DOMAIN>_CODE`, consumer/access schemas, or plugin schemas is not permitted.

---

## Dynamic SQL rules

Dynamic SQL must be avoided unless there is a clear need.

Where dynamic SQL is required:

- Use bind variables.
- Do not concatenate untrusted values into SQL.
- Do not let the LLM provide executable SQL.
- Do not let plugins provide executable SQL unless the capability has an explicit reviewed design.
- Validate object names against allowlists when dynamic object references are unavoidable.
- Do not use dynamic SQL to bypass TAPI or API-layer rules.

Security rules for generated SQL are defined in:

```text
docs/agent-guardrails/60-security-and-risk.md
```

## Database object creation scripts

As a rule, each database object or constraint should have its own script.

Exceptions:

- comments for a single table or view may be grouped in one comment script
- annotations for a single table may be grouped in one annotation script
- grants for a single object may be grouped where this improves maintainability
- seed data may be grouped by functional area where appropriate
- install driver scripts may call other scripts

Script file name stems should reflect the object being created.

Table annotation scripts must be named after the table they annotate, using
`<table-name>.sql`.

Script files must use the `.sql` extension unless the repository already uses a different convention for package specs or bodies.

Examples:

```text
table/messages.sql
constraint_pk/messages_pk.sql
constraint_fk/messages_conversations_fk1.sql
constraint_ck/messages_role_ck.sql
index/messages_conversation_idx1.sql
comment/messages_comments.sql
annotation/messages.sql
package_spec/messages_tapi.pks
package_body/messages_tapi.pkb
```

Use the repository's existing file extension conventions where already established.

## Naming principles

Relational object names use `_` as the separator.

Object names should be meaningful and specific.

Avoid generic names.

Do not use Oracle reserved words or keywords as object names.

Do not use quoted identifiers.

Agents must check:

```text
docs/agent-guardrails/table-abbreviations.csv
```

before creating or renaming tables, or database objects that depend on table
abbreviations.

If a table abbreviation is missing for a new table, propose and add it to
`table-abbreviations.csv` in the same change as the table DDL. Do this even
when constraints and indexes are deferred.

Do not use generic names such as:

```text
pk_<table>
fk_<table>_<parent>
idx1
constraint1
```

unless explicitly instructed.

---

## Renaming

Do not rename existing objects to improve naming consistency.

Existing object names take precedence over new naming standards.

Renaming is only permitted when explicitly instructed.

Do not introduce synonyms or duplicate objects as a workaround for renaming.

If a name is inconsistent but already used, preserve it unless the user explicitly requests a rename.

---

## Table naming

Tables should use meaningful singular names unless an existing project pattern requires plural names.

Use the existing project naming style.

Where table prefixes or domain prefixes are already established, follow the existing project convention.

Examples:

```text
message
conversation
plugin_registry
plugin_capability
```

Avoid vague names:

```text
data
items
records
details
```

---

## Table abbreviations

Table abbreviations identify table entities and reserve the naming prefix used
by later constraints, indexes, triggers, TAPIs, and related database objects.

Rules:

- Reuse the abbreviation from `table-abbreviations.csv`.
- Add a table abbreviation whenever a new table is created.
- Do not invent abbreviations silently.
- Do not treat a mechanically derivable abbreviation as automatically approved.
- Construct new proposed abbreviations as `<namespace>_<alias>`.
- Use the characters at the start of the table name before the first underscore
  as the namespace, usually 3 characters.
- Derive the alias from the rest of the table name after the namespace.
- Keep the alias to a maximum of 6 characters.
- If needed for uniqueness within the schema, use a number as the final alias
  character.
- Keep abbreviations stable once used.
- Do not change abbreviations merely to improve readability.
- When asked to work on or review scripts for a table with no registered
  abbreviation, prompt the user with the option to add one.
- If the user supplies an abbreviation, update `table-abbreviations.csv` in the
  same change before using it in dependent object names.
- If an abbreviation is missing and not obvious from the documented convention,
  propose it and ask before adding it.
- In a new or empty abbreviation register, treat the abbreviation as not obvious
  unless the user has already supplied or approved it.
- Do not defer the abbreviation entry just because constraints or indexes are
  not being created yet.

Example mapping:

```text
cfg_config_apps = cfg_apps
cfg_configuration_items = cfg_config
cfg_configuration_types = cfg_confi1
```

In `cfg_config_apps`, `cfg` is the namespace and `apps` is the shortened alias
for the remaining table name. In `cfg_configuration_types`, `cfg_confi1` uses a
numeric final alias character only if required to stay unique within the schema.

Actual abbreviations must come from the CSV file, not from guesses.

## Column naming

Column names should be meaningful and specific.

Use `_` as the separator.

Avoid abbreviated column name units unless the abbreviation is already standard in the project.

Avoid trailing numeric suffixes where a clearer name exists.

Good:

```text
address_line1
created_on
message_type
conversation_id
```

Avoid:

```text
addr_1
dt
typ
val
```

## Standard columns

Mutable project-owned tables should normally include these standard columns:

| Column | Datatype | Nullable | Default | Purpose |
|---|---|---:|---|---|
| `created_on` | `timestamp` | No | `systimestamp` | Database timestamp when the row was created. |
| `created_by` | `varchar2(128 char)` | No | current application/session user expression | User, job, or process that created the row. |
| `updated_on` | `timestamp` | Yes | none | Database timestamp when the row was last updated. |
| `updated_by` | `varchar2(128 char)` | Yes | none | User, job, or process that last updated the row. |
| `row_version` | `number` | No | `1` | Optimistic locking value incremented on update. |

Recommended `created_by` expression:

```sql
created_by varchar2(128 char) default coalesce(
             sys_context('userenv', 'client_identifier'),
             sys_context('userenv', 'proxy_user'),
             sys_context('userenv', 'session_user'),
             user
           ) not null
```

Projects may substitute a reviewed equivalent current-user expression where needed.

Recommended `created_on` expression:

```sql
created_on timestamp default systimestamp not null
```

Recommended `row_version` expression:

```sql
row_version number default 1 not null
```

Do not remove standard columns from existing tables without explicit approval.

Do not remove `row_version` from mutable tables unless the table is explicitly externally owned or append-only and the exception is documented.

---

## Soft delete columns

Where soft delete is required, use explicit soft delete columns.

Recommended optional columns:

| Column | Purpose |
|---|---|
| `deleted_on` | Timestamp when the row was soft deleted. |
| `deleted_by` | User, job, or process that soft deleted the row. |
| `deletion_reason` | Optional reason for the soft deletion. |

API views should normally hide soft-deleted rows unless the view is explicitly intended to expose them.

TAPI packages should own the soft-delete behaviour.

Do not implement soft delete ad hoc in application-facing SQL.

---

## Datatypes

Use Oracle datatypes deliberately.

Rules:

- Do not use `varchar`.
- Do not use `char` unless there is a specific fixed-width requirement.
- Use `varchar2`.
- Specify character semantics for text columns, for example `varchar2(100 char)`.
- Use `number` for numeric values unless a more precise type is required.
- Use identity columns for surrogate numeric primary keys where already used.
- Use `timestamp` or `timestamp with time zone` where time precision matters.
- Use `date` only where date-only or legacy semantics are intended.
- Use native `json` datatype where supported by the target Oracle version.
- Use `clob` plus `is json` only where native `json` is not available or not appropriate.
- Use `xmltype` for XML data.

Examples:

```sql
message_type varchar2(30 char) not null
content json not null
tokens_used number
created_on timestamp default systimestamp not null
```

---

## Default values

Prefer `default on null` where a value should be supplied even when the insert explicitly provides `null`.

Example:

```sql
message_id number generated by default on null as identity not null
```

For audit timestamps, use the existing project convention:

```sql
created_on timestamp default systimestamp not null
```

Do not add defaults that hide missing required business data.

---

## Yes/No columns

Simple yes/no columns should normally use `Y` and `N`.

Column names should normally end with `_yn`.

Example:

```sql
active_yn varchar2(1 char) default 'Y' not null
```

Add a check constraint:

```sql
alter table <DOMAIN>_CORE.example_table add constraint exmp_active_ck1
  check (active_yn in ('Y', 'N'));
```

Do not use `Yes`/`No`, `true`/`false`, or `1`/`0` for yes/no columns unless an existing integration requires it.

---

## Code columns

Columns used for codes should normally be uppercase and constrained.

Recommended pattern:

```sql
alter table <DOMAIN>_CORE.example_table add constraint exmp_code_ck1
  check (regexp_like(example_code, '^[A-Z0-9_]+$'));
```

Use code columns for stable natural identifiers where appropriate.

Do not use unconstrained free text where a controlled code is required.

---

## Primary keys

Primary keys should be defined in separate `alter table` statements.

Do not define primary keys inline in the `create table` statement.

Preferred pattern:

```sql
create table <DOMAIN>_CORE.messages
(
  message_id number generated by default on null as identity not null,
  ...
);

alter table <DOMAIN>_CORE.messages add constraint msg_pk
  primary key (message_id);
```

Primary key constraint naming:

```text
{table_abbr}_pk
```

---

## Surrogate keys

Surrogate key columns should normally use the table name or approved abbreviation plus `_id`.

Examples:

```text
message_id
conversation_id
plugin_id
```

Where existing project tables already use a different but clear style, preserve the existing style.

Do not rename existing surrogate key columns to satisfy a new naming preference.

Surrogate keys should remain identity columns where already used.

---

## Foreign keys

Foreign key columns should normally reflect the referenced table and column.

Preferred simple pattern:

```text
{referenced_table_singular}_id
```

Examples:

```text
conversation_id
plugin_id
llm_id
```

Foreign key constraint naming:

```text
{child_abbr}_{parent_abbr}_fk{seq_nr}
```

For multiple foreign keys from the same child table to the same parent table, increment the sequence number.

For self-referencing foreign keys, use the same abbreviation for child and parent.

---

## Cross-schema foreign keys

Cross-schema referential integrity requires `grant references`.

Example:

```sql
grant references on <PARENT_SCHEMA>.parent_table to <CHILD_SCHEMA>;
```

The grant must be applied before the foreign key constraint is created.

Use cross-schema foreign keys deliberately.

Do not add cross-schema referential integrity without considering install order, ownership, and deployment complexity.

---

## Constraints

Constraint naming patterns:

| Constraint type | Pattern |
|---|---|
| Primary key | `{table_abbr}_pk` |
| Foreign key | `{child_abbr}_{parent_abbr}_fk{seq_nr}` |
| Check constraint | `{table_abbr}_ck{seq_nr}` |
| Unique constraint | `{table_abbr}_uk{seq_nr}` |
| Column check constraint | `{table_abbr}_ck{seq_nr}` |

Do not explicitly name `not null` constraints unless an existing project pattern requires it.

Use constraints to enforce data quality wherever possible.

Do not rely only on application logic for data quality.

---

## Indexes

Primary keys and unique constraints should be supported by indexes.

Foreign keys should normally be indexed where they are used in joins or where child-row locking could matter.

Index naming patterns:

| Index type | Pattern |
|---|---|
| Primary key index | same as primary key constraint |
| Unique index | `{unique_constraint}_idx` |
| Foreign key index | `{foreign_key_constraint}_idx` |
| Function-based index | `{table_abbr}_fidx{seq_nr}` |
| Bitmap index | `{table_abbr}_bidx{seq_nr}` |
| Other index | `{table_abbr}_idx{seq_nr}` |

### Concatenated indexes

When creating concatenated indexes, choose column order deliberately.

The most selective and commonly filtered leading columns should normally come first.

Do not create concatenated indexes without considering how the leading columns will be used.

### Function-based indexes

Use function-based indexes carefully.

Avoid function-based indexes that depend on application package functions unless there is a reviewed design.

Common acceptable example:

```sql
create index exmp_fidx1 on <DOMAIN>_CORE.example_table (upper(example_code));
```

### Bitmap indexes

Bitmap indexes are normally unsuitable for OLTP-style tables.

Use bitmap indexes only where the table and workload justify them.

---

## Views

Regular views must end with `_v`.

Do not prefix views with `v_`.

API views in `<DOMAIN>_API` should normally be simple pass-through projections over one `<DOMAIN>_CORE` table.

Business views in `<DOMAIN>_CODE` may include joins, calculations, and business rules.

Preferred pattern:

```sql
create or replace force view <DOMAIN>_API.messages_v as
select message_id
     , conversation_id
     , message_type
     , turn_index
     , role
     , content
     , tokens_used
     , meta
     , llm_id
     , created_on
     , created_by
     , updated_on
     , updated_by
     , row_version
from <DOMAIN>_CORE.messages;
```

Views should not use owner-qualified object names where private synonyms or direct ownership already provide the approved access path.

Do not embed application-specific logic in API views.

---

## Materialized views

Materialized views must end with `_mv`.

Do not prefix materialized views with `mv_`.

Materialized views belong in `<DOMAIN>_CODE` unless there is an explicitly approved reason to place them elsewhere.

Use materialized views for deliberate performance, aggregation, or pre-computation needs.

Do not use materialized views to bypass API access rules.

---

## Triggers

Table triggers that maintain audit columns, row versions, or API-layer data rules belong in `<DOMAIN>_API`.

Trigger names should include the table name and trigger timing/action.

Suggested suffixes:

| Timing/action | Suffix |
|---|---|
| Before insert row | `_bi` |
| Before update row | `_bu` |
| Before insert or update row | `_biu` |
| After insert row | `_ai` |
| After update row | `_au` |
| After insert, update, delete row | `_aiud` |
| Compound trigger | `_ct` |

Follow the existing repository naming pattern where one exists.

Do not create triggers in `<DOMAIN>_CORE` unless explicitly approved.

### Audit User and Row Version Maintenance

Tables that use standard audit columns or `row_version` must have a before-row trigger in `<DOMAIN>_API` when those values cannot be maintained safely by column defaults alone.

Use column defaults for immutable creation values where practical. Use triggers to maintain update audit values and optimistic locking values consistently.

Recommended trigger behaviour:

- On insert, initialise `created_on`, `created_by`, and `row_version` where those values are not already enforced by defaults.
- On update, preserve `created_on` and `created_by`.
- On update, set `updated_on` and `updated_by`.
- On update, increment `row_version`.
- Do not allow application code to bypass `row_version` maintenance.

Example:

```sql
--liquibase formatted sql

--changeset <login_name>:<table_abbr>_biu_create stripComments:false endDelimiter:/ runOnChange:true
create or replace trigger <DOMAIN>_API.<table_abbr>_biu
before insert or update on <DOMAIN>_CORE.<table_name>
for each row
begin
  if inserting
  then
    :new.created_on := coalesce(:new.created_on, systimestamp);
    :new.created_by := coalesce(
      :new.created_by,
      sys_context('apex$session', 'app_user'),
      sys_context('userenv', 'proxy_user'),
      sys_context('userenv', 'session_user'),
      user
    );
    :new.row_version := coalesce(:new.row_version, 1);
  elsif updating
  then
    :new.created_on := :old.created_on;
    :new.created_by := :old.created_by;
    :new.updated_on := systimestamp;
    :new.updated_by := coalesce(
      sys_context('apex$session', 'app_user'),
      sys_context('userenv', 'proxy_user'),
      sys_context('userenv', 'session_user'),
      user
    );
    :new.row_version := nvl(:old.row_version, 0) + 1;
  end if;
end;
/

--rollback drop trigger <DOMAIN>_API.<table_abbr>_biu;
```

## Packages

Package names should be meaningful and singular.

TAPI packages belong in `<DOMAIN>_API`.

Business logic packages belong in `<DOMAIN>_CODE`.

Consumer-specific packages belong in the relevant consumer/access schema only where needed.

Standalone procedures and standalone functions are discouraged.

Prefer packages for related functionality.

## Sequences

Use identity columns for surrogate keys where already used.

Do not introduce sequences for new surrogate keys unless there is a specific reason.

Where sequences are required, use a clear name based on the owning object.

Example:

```text
example_table_seq
```

## Comments

Tables and columns should have comments.

Comments should explain purpose, not merely repeat the name.

Bad:

```sql
comment on table <DOMAIN>_CORE.messages is
  'Stores messages.';
```

Better:

```sql
comment on table <DOMAIN>_CORE.messages is
  'Stores persisted interaction events used to reconstruct message history and build runtime context.';
```

Column comments should describe business meaning, expected values, and important constraints.

Comments for a table or view may be grouped in one script.

## Annotations

Oracle Database 26ai database object annotations are table-scoped metadata
scripts in this scaffold. They refer to the Oracle Database annotation feature
introduced with Oracle Database 26ai, not Liquibase formatted SQL control
comments.

Annotations are structured schema metadata. They must be deliberate,
consistent, and usable by tools.

Use annotations for machine-readable metadata about database objects and
columns, such as display names, UI behaviour, sensitivity classification,
semantic role, business glossary references, lifecycle flags, API exposure, or
tooling hints.

Do not use annotations as long-form documentation. Use comments for prose
descriptions and annotations for structured metadata.

Place table and column annotations for a table in that table's annotation script:

```text
annotation/<table-name>.sql
```

Annotation scripts belong in the same schema directory as the annotated table
unless the repository documents a different owner or install convention.

All annotations related to one table should live in one script.

Annotation scripts must be installed after the related comment scripts.

When creating or changing tables or columns in an Oracle Database 26ai target,
explicitly decide whether database object annotations are required. If the
repository has an annotation vocabulary or nearby examples, add or update the
annotation script in the same change. If the vocabulary or required values are
missing, stop and ask for the source of truth instead of silently omitting
annotations.

Annotation names must come from a documented project vocabulary unless the task
explicitly extends that vocabulary.

Annotation values must be minimal and parseable.

Prefer:

- controlled labels
- stable identifiers for glossary, policy, or domain references
- consistent boolean or enum-like values
- scalar values or compact structured values that downstream tools can parse

Avoid:

- vague prose blobs
- raw model output
- ad-hoc labels, names, or value shapes
- multiple unrelated meanings in one annotation value

Do not fabricate business terms, sensitivity classifications, policy
references, lineage, ownership metadata, confidence scores, source references,
or provenance.

If the requested annotation content is not directly supported by the table DDL,
existing comments, established project annotation vocabulary, nearby examples,
or explicit user or project requirements, stop and ask for the missing source of
truth before inventing annotation values.

Do not ask questions for purely mechanical annotation placement or formatting
when the target table, annotation names, and annotation values are already
specified.

When changing annotation names, value formats, or meanings, check downstream
consumers and update validation, tests, and documentation where applicable.

## Message history and context columns

If the project implements a persisted message history store, the behavioural meaning of columns such as `<MESSAGE_HISTORY_TABLE>.role` and `<MESSAGE_HISTORY_TABLE>.message_type` must be explicitly documented by the project.

The database layer may enforce allowed values and allowed combinations.

The semantic rules for context assembly, tool-result visibility, summaries, stale context, and provenance must be documented in the project's architecture or security guardrails.

Do not add message roles or message types ad hoc from application code.

---

## JSON columns

Use native `json` columns where the target Oracle version supports them.

JSON content must have a documented structure.

Do not use JSON as a dumping ground for data that should be relational.

Use JSON for:

- model payloads
- plugin result payloads
- flexible metadata
- external service payloads
- context fragments

Do not use JSON to avoid modelling stable relational data.

Do not store secrets in JSON content or metadata.

Security rules for secrets are defined in:

```text
docs/agent-guardrails/60-security-and-risk.md
```

---

## Seed data

Seed data scripts belong in `seed_data`.

Seed data should be deterministic.

Use stable codes rather than generated IDs where possible.

Seed data scripts should be repeatable where practical.

Do not write seed data that depends on an uncontrolled current environment state unless explicitly required.

---

## Installation scripts

Installation scripts should preserve schema boundaries.

Install scripts must not:

- drop unrelated objects
- grant broad privileges
- connect as highly privileged users unless explicitly required
- hide errors
- silently ignore failed DDL
- change ownership boundaries
- remove constraints without approval

Install scripts should be clear, ordered, and repeatable.

## Agent rules

When generating or modifying database code, agents must:

1. Read this document first.
2. Check and update `table-abbreviations.csv` before creating tables and before
   naming constraints or indexes.
3. Preserve existing object names unless explicitly told to rename.
4. Preserve identity surrogate keys where already used.
5. Preserve standard audit and row version columns.
6. Preserve schema boundaries.
7. Use TAPIs for DML from `<DOMAIN>_CODE`.
8. Avoid direct access to `<DOMAIN>_CORE` outside the approved API layer.
9. Use lowercase SQL and PL/SQL reserved words.
10. Use 2-space indentation.
11. Add or update comments and Oracle Database 26ai database object annotations
    where new tables or columns are created; if annotation names or values are
    not known, ask for the project vocabulary or source of truth.
12. Add constraints for controlled values.
13. Add tests or install verification where appropriate.
14. Regenerate or update schema dependency scripts when source DDL changes affect generated views or grants.
15. Stop and ask before introducing a new schema, grant pattern, or naming convention.

## Final rule

The database model must preserve this separation:

```text
<DOMAIN>_CORE            = data ownership
<DOMAIN>_API             = controlled table access
<DOMAIN>_CODE            = business logic
<DOMAIN>_<CONSUMER>_PUB  = consumer-specific access
<PLUGIN_SCHEMA>          = plugin-private storage
```

Any code that bypasses this separation is a design defect.
