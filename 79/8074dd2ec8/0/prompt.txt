You are working in the Orac repository.

Use high reasoning.

Goal:
Wire the existing PluginServiceManager into the Orac runtime lifecycle so service and hybrid plugin manifests have real supervised startup and shutdown behaviour.

This follows the architecture audit and the recent protocol/schema hardening work.

Scope:
- Focus only on plugin service lifecycle wiring.
- Do not implement Home Assistant device control.
- Do not add plugin execution policy yet.
- Do not refactor the Orac controller broadly.
- Do not introduce an LLM provider registry.
- Do not change protocol schema or display events.
- Do not tidy unrelated files.
- Preserve existing on-demand plugin routing behaviour.

Background:
The audit found that PluginServiceManager exists and has tests, and manifests such as plugins/home_assistant.json can declare runtime.mode as service or hybrid with start_policy auto. However, Orac initialises PluginManager and PluginRouter but does not appear to wire PluginServiceManager into live startup/shutdown.

Goal behaviour:
1. At Orac startup or plugin refresh, service and hybrid manifests are registered with PluginServiceManager.
2. Plugins with service start_policy auto are started under PluginServiceManager supervision.
3. Manual service plugins are discovered but not started.
4. Disabled, invalid, or dependency-failed service plugins are not started.
5. Orac shutdown stops all managed service plugins cleanly.
6. Existing on-demand plugin behaviour remains unchanged.
7. Home Assistant may remain scaffold/experimental, but if its manifest says enabled and auto-start, the runtime must either:
   - start only the safe placeholder service under supervision, or
   - explicitly skip it because it is marked scaffold/experimental/unsupported.
   Do not silently ignore it.

Phase 0 - Inspect before editing:
Read:
- architecture-qa-results.md
- src/controller/orac.py
- src/model/plugin_service_manager.py
- src/model/plugin_routing/manager.py
- src/model/plugin_router.py
- src/model/plugin_runtime.py
- plugins/home_assistant.json
- plugins/README.md
- tests/test_plugin_service_manager.py
- any existing Orac startup/shutdown tests

Before editing, summarise:
- where plugin manifests are currently discovered
- where PluginRouter is initialised
- where Orac startup/refresh/shutdown hooks exist
- the minimal integration point for PluginServiceManager

Phase 1 - Runtime lifecycle wiring:
Implement the smallest safe integration:
- Add a PluginServiceManager instance to the Orac runtime where plugin routing is initialised.
- Register discovered service/hybrid manifests with it.
- Start auto services at the appropriate lifecycle point.
- Stop all managed services during shutdown/cleanup.
- Make lifecycle logging clear:
  - discovered
  - skipped
  - started
  - failed
  - stopped

Avoid broad controller extraction. A small helper method is acceptable if it keeps the controller readable.

Phase 2 - Manifest/runtime compatibility:
Ensure service/hybrid mode handling is explicit:
- on_demand plugins remain routed as before
- service plugins are managed by PluginServiceManager
- hybrid plugins can be both routable and service-managed
- disabled plugins are not started
- missing dependency plugins are skipped with clear logging

If Home Assistant is still scaffold-only, either:
- mark it explicitly as scaffold/experimental and skip real auto-start, or
- start only its safe placeholder service if that is already implemented and harmless

Do not implement actual HA websocket/device control.

Phase 3 - Tests:
Add or update tests covering:
1. Orac initialises PluginServiceManager.
2. Service and hybrid manifests are registered during plugin setup/refresh.
3. auto-start service plugins are started.
4. manual service plugins are discovered but not started.

6. dependency-failed service plugins are not started.
7. shutdown stops started services.

Prefer testing via the public Orac lifecycle where possible. If private methods must be used because no public lifecycle seam exists yet, mark those tests as interim and keep them small.
